Work Blocks

Shippable units of work

How Work Blocks Work

A work block is a shippable unit of work -- a group of issues that together produce something you can deploy or demo. Think of it as a goal-driven sprint without time constraints.

Proposed

You or the CEO proposes a block with a title and goal. It needs human approval to become active.

Active

Only one block can be active at a time. Assign issues to it, and agents work toward the block's goal. The CEO prioritizes block issues over standalone work.

Ready

When all issues are done (or the CEO marks it ready), the block awaits your review. Check the metrics, review the work, then ship or send it back.

Shipped

You approved the block for deployment. Deployment happens outside SecondOrder -- agents produce the work, you ship it. A new block can now become active.

Key rules: One active block at a time. Blocks are optional -- issues work fine without them. No time boxing -- a block is done when all its issues are done. The CEO can assign and unassign issues from the active block.

Metrics: Each block tracks total cost, run count, issues planned vs completed, and cycle time. Use these to understand delivery velocity and cost efficiency across iterations.
